Docker 开发容器图形化 AI 工作站配置教程

发表于2026-05-24|更新于2026-05-24|技术教程
|浏览量:

背景

本文记录一套完整的远程 Docker 开发容器图形化配置流程。

目标是在 Windows 上通过 VS Code Remote-SSH 连接服务器 Docker 容器，并在容器中长期使用 AI Toolbox、OpenCode、Claude Code、Codex 等工具。

这篇文章是上一篇《Windows 远程开发服务器目录教程，SSHFS、Docker容器、VS Code和Conda》的后续。上一篇主要解决 Windows 如何连接远程服务器目录和 Docker 容器。本文继续解决容器内部如何做成一个可重建、可持久化、可图形化使用的 AI 工作环境。

最终效果如下。

环境约定

为避免泄露真实信息，本文统一使用以下脱敏示例。

项目 示例值 说明
服务器 IP 192.168.1.100 替换为你的服务器 IP
宿主机用户 serveruser 替换为你的服务器用户
容器用户 devuser 替换为你的容器用户
Docker 项目目录 /data/docker-dev 存放 docker-compose.yml 和 Dockerfile
宿主机数据目录 /data/projects/all 挂载到容器的工作区
容器工作区 /workspace/all 容器内主要工作目录
容器 SSH 端口 2202 宿主机映射到容器 22 端口
Mihomo 代理端口 59000 宿主机代理端口示例
noVNC 端口 6080 浏览器访问远程桌面
VNC 端口 5901 TigerVNC 默认 :1 会话端口

真实公网 IP、内网 IP、用户名、私钥、公钥、SSH key 指纹、完整服务器路径不要写进公开博客。

目标

本文完成以下配置。

  • Windows 通过 SSH key 免密登录 Docker 容器
  • 宿主机通过 SSH key 免密登录 Docker 容器
  • 容器重建后 SSH host key 不变化，避免反复出现 REMOTE HOST IDENTIFICATION HAS CHANGED
  • Docker 容器运行时和构建时都能使用宿主机 Mihomo 代理
  • ~/.ssh、~/.config、~/.local、~/.cache、~/.vnc 等关键目录持久化
  • Dockerfile 固化 XFCE、TigerVNC、noVNC、WebKitGTK 等图形运行依赖
  • 通过 VS Code 转发 6080 端口，在浏览器中访问容器桌面
  • 在容器图形桌面中运行 AI Toolbox
  • AI Toolbox 能识别容器里的 OpenCode 配置
  • 容器重建后仍可恢复桌面和 AI Toolbox

总结

这套方案的关键是把 Docker 容器拆成两部分理解。

系统依赖放进镜像，包括 SSH 服务、XFCE、TigerVNC、noVNC、WebKitGTK 和 AI Toolbox 运行库。

用户数据放进持久化挂载，包括 SSH key、SSH host key、OpenCode 配置、AI Toolbox 数据、VNC 密码和启动脚本。

这样容器即使重建，也可以保留登录状态、配置目录、图形桌面配置和 AI Toolbox 数据。

最终得到的是一个可以通过 Windows VS Code 连接、可以跑命令、可以开图形桌面、可以启动 AI Toolbox 的远程 Docker AI 工作站。
